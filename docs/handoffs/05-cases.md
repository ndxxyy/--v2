# 05 医案整理交接

## 完成内容

- 将医案列表与详情路由从任务 01 骨架替换为完整业务页面，沿用“青瓷编目”的颜色、字体、栅格、细分隔线、焦点和触控规范。
- 建立内科、妇科、儿科、针灸、调理随笔、其他临证记录六个分类，提供简体中文、繁体中文和英文名称及中性分类说明；分类不暗示当前已有文章、资质、执业范围或疗效。
- 保持 `src/data/cases.ts` 为空，没有从旧站迁移、改写或编造任何医案；文件注释明确患者资料、审核证明和可识别信息不得进入仓库。
- 新增医案公开选择器，公开记录必须同时满足：
  - 发布状态为 `published`。
  - 匿名化审核为 `approved`。
  - 患者授权为 `approved`。
  - 编辑审核为 `approved`。
  - 当前语言有非空标题与正文。
  - 存在有效发布日期。
  - 正文不含尚未建立安全资源管线的图片块。
- 选择器只向页面返回标题、公开摘要、分类、发布日期、正文和公开 URL，不返回患者资料、审核记录或证据引用。
- 医案目录有真实公开记录时才显示分类筛选、当前条件与真实数量；当前数据为空，因此不渲染无效筛选、空列表数字、假标题或假阅读入口。
- 空状态明确说明栏目用途、四重审核门槛、发布日期要求和隐私边界；不展示患者身份、原始检查单、可识别面部、完整出生日期、患者见证或疗效承诺。
- 分类在空数据状态下作为“分类方法”展示，不伪装成可点击筛选或文章数量。
- 详情页实现文本、标题、列表和三类合规提示块渲染；未公开、不存在、缺少当前语言正文、缺少发布日期或包含未受控图片块的记录均返回 404。
- 详情页底部只保留“了解作者”“查看就诊说明”“更多医案”三个克制入口，并固定展示不提供远程诊断、不替代面诊、不构成疗效承诺的医学信息边界。
- 使用项目级 Impeccable 上下文与品牌规范完成设计约束；自动 Hook 继续保持关闭。

## 遗留问题

- 当前没有用户提供并完成匿名化、患者授权、编辑审核与发布日期确认的真实医案，因此公开数据保持为空，无法对真实长文、真实摘要和真实发布日期进行内容级视觉复核。
- 医案正文图片暂不公开。现有通用内容模型只有 `imageId`，但任务 05 范围内没有经审核的医案图片清单、固定宽高、替代文本或安全派生资源管线；选择器会拒绝包含图片块的记录。
- 任务 07 仍需完成最终三语言医学术语审校、Metadata、canonical、hreflang 和医案结构化数据；不得放宽本任务的公开筛选条件。
- 关于作者与联系页面的最终真实内容属于任务 06；本任务只使用已冻结的站内路径，不写入或推断作者资质、机构、联系方式或接诊信息。
- 项目目录没有 Git 元数据，无法用 `git status` / `git diff` 识别并发修改；本任务按文件所有权只写入 05 独占路径。

## 后续任务注意事项

1. 新增医案前必须取得真实资料，并在仓库外完成匿名化、患者网站公开授权与编辑审核；审核证明不得写入公开仓库。
2. `src/data/cases.ts` 不得加入患者姓名、手机号、身份证、详细地址、完整出生日期、原始检查单、面部图像、真实病案号、患者见证、治愈结论或疗效评分。
3. 新记录必须提供真实发布日期和当前语言已审核正文；只完成 `publication.status` 不会通过任务 05 的公开选择器。
4. 若未来确需公开医案配图，应先由架构任务建立独立、可审核的图片数据与派生资源管线，再调整选择器；不得直接引用原始检查单、患者面部或母版路径。
5. 任务 03 首页选择器目前沿用冻结的四重审核条件；任务 07/08 若统一规则，应复用 `src/features/cases` 的公开入口或保持同等严格，不能让首页绕过发布日期和安全正文检查。
6. 任务 07 添加结构化数据时只读取公开选择器返回的字段，不得序列化审核记录、授权记录或证据引用。
7. 任务 08 应在所有并行构建结束后重新运行一次全量正式构建，并在真实医案资料到位后补做详情页长文、三语言文本扩展和 320px 复核。
