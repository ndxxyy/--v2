# 中药图谱小批次更新模板

本模板用于任务 09 的单味或小批次中药更新。每个工作分支只处理一个批次，推荐 1–5 味。复制“批次记录”和“单味记录”后填写真实资料；`待填写`、`未提供`、`待审核` 只能表示工作状态，绝不能写入公开数据冒充内容。

## 1. 不可跳过的规则

1. 从最新、干净的 `main` 创建独立分支；框架分支为 `content/herbs-v1.1`，后续批次按顺序使用独立批次分支。
2. 旧网站只读；母版、密钥、追加式登记表、高清授权交付文件、版权证明和医学审校证据不得提交 Git，也不得放入 `public`。
3. 每味中药必须有真实且唯一的稳定 `id` 与 ASCII 小写连字符 `slug`；作品编号须提供真实值或明确标记缺失。新增编号必须经用户确认，不得猜测下一个编号或发明分类缩写。
4. 简体、繁体、英文标题和简介必须分别审核；任一必需语言未审核时，作品保持 `draft`，不得以自动翻译冒充审校结果。
5. 不从图片识别或推断性味、归经、功效、主治、剂量或疗效。需要把医学内容改为网页结构化文字时，必须先扩展 `Work` 模型并同步 `docs/content-model.md`。
6. 图片来源、版权状态、公开许可和明水印策略必须明确。已带印章或网状标识的图片保留原标识，不重复叠加；没有可确认标识或策略时停止发布并请求用户决定。
7. 所有公开变体必须由脚本生成并嵌入 `xqy-dct-qim-v2` 盲水印。沿用现有密钥与追加式登记表，不得为批次新建密钥，不得删除历史指纹。
8. 数量只从版本化审核清单中的显式资产和变体角色计算。未登记、重复、缺图、路径冲突、哈希不符或水印策略未审核时必须失败。
9. 只有 `published` 会进入公开选择器；“已录入”“已审校”或 `review` 都不等于获准公开。
10. 默认仅本地工作。没有对应用户指令时，不推送、不创建 PR、不调用 Cloudflare、不合并 `main`、不创建 Release、不部署正式 Worker。

## 4. 批次实施顺序

1. 将来图复制到受控本地目录；原始文件不进入 `public`。
2. 完成逐味身份、三语言、医学来源、版权、公开许可和母版检查。
3. 将通过审核的母版导入 `content-assets/masters/works-watermarked/herbs/<分类>/<slug>.<格式>`。
4. 更新版本化显式资产清单；不能靠目录扫描信任新增文件。
5. 更新 `src/data/works.ts` 和 `src/data/image-groups.ts`；没有真实资料的字段省略，作品先保持 `draft`。
6. 用受保护母版运行图片生成器；优先只生成本批次资产，未变化资产尽量不重新编码。生成目录和 `src/data/atlas-image-assets.generated.ts` 只能由脚本更新，输出必须先进入暂存目录，全部通过后再原子替换。
7. 先验证本批次，再进行全库路径、哈希、明水印描述和盲水印验证；任一失败时保留上一版公开资源。
8. 检查本草数量、搜索、详情、相关作品、首页统计和精选是否与真实数据一致；未经用户确认不改变首页精选。
9. 分别检查三语言页面、Metadata、静态路径和 sitemap；未审核语言不得公开。
10. 把真实变化写入 `CHANGELOG.md`，把完整验证和发布状态写入 `docs/handoffs/09-version-update.md`。

## 7. 分阶段权限闸门

### 阶段 1：默认，仅本地

- [ ] 只修改本地分支并运行本地检查。
- [ ] 没有推送分支、创建 PR 或调用 Cloudflare。
- [ ] 没有合并 `main`、创建标签或 Release。
- [ ] 没有运行任何生产 `deploy` 命令。
- [ ] 对外状态明确写为“仅本地，尚未上线”。

### 阶段 2：用户明确要求“创建预览，不上线”后

- [ ] 已记录用户的明确预览授权。
- [ ] 只推送当前工作分支并创建 PR。
- [ ] 只创建 Cloudflare 非生产 Preview Version；使用项目确认的 `upload` 流程，禁止使用 `deploy`。
- [ ] 已验证 Preview URL，并停止等待用户确认。
- [ ] 正式 `main`、正式 Worker 和固定正式 URL 未发生变化。

### 阶段 3：用户明确要求 `确认发布 <版本号>` 后

- [ ] 已记录用户明确确认的版本号；没有沿用旧批次授权。
- [ ] 已重新同步并检查 PR、数量、三语言、版权、公开许可和全库水印结果。
- [ ] 已确认 `main` 是唯一 Production branch。
- [ ] 合并后等待正式 Worker 自动部署，并检查固定正式 URL。
- [ ] 标签与 Release 只使用真实确认版本；异常时回退上一 Cloudflare 版本并撤销 PR，不强推或重置共享 `main`。
